# CI/CD Performance Testing Integration\n\nComprehensive guide for integrating performance tests into GitHub Actions CI/CD pipeline with automated monitoring, alerting, and reporting.\n\n## Overview\n\nThe CI/CD pipeline automatically runs performance tests on every commit and pull request, validating SLA compliance and detecting regressions. Results are reported as PR comments, stored for historical analysis, and trigger alerts on violations.\n\n## Workflow Configuration\n\n### GitHub Actions Workflow\n\nLocation: `.github/workflows/performance-tests.yml`\n\n**Triggers**:\n- Push to `main` and `develop` branches\n- Pull requests to `main` and `develop` branches\n- Daily schedule at 2 AM UTC\n\n**Jobs**:\n1. **Setup**: Install dependencies, build application\n2. **Database**: Start MySQL service\n3. **Tests**: Run baseline and stress tests\n4. **Validation**: Check SLA compliance\n5. **Comparison**: Detect regressions vs baseline\n6. **Reporting**: Generate HTML reports and PR comments\n7. **Alerts**: Notify Slack on failures\n\n## Setup Instructions\n\n### 1. GitHub Secrets Configuration\n\nAdd the following secrets to your GitHub repository:\n\n**Settings → Secrets and variables → Actions**\n\n```\nSLACK_WEBHOOK_URL\n  Description: Slack webhook for notifications\n  Value: https://hooks.slack.com/services/YOUR/WEBHOOK/URL\n\nMETRICS_DB_URL\n  Description: Database URL for storing metrics\n  Value: mysql://user:password@host:3306/metrics_db\n```\n\n### 2. Slack Webhook Setup\n\n1. Go to your Slack workspace\n2. Create an Incoming Webhook: https://api.slack.com/messaging/webhooks\n3. Copy the webhook URL\n4. Add as `SLACK_WEBHOOK_URL` secret in GitHub\n\n### 3. Metrics Database Setup (Optional)\n\nFor historical metrics tracking, set up a database:\n\n```sql\nCREATE TABLE performance_metrics (\n  id INT AUTO_INCREMENT PRIMARY KEY,\n  commit_sha VARCHAR(40) NOT NULL,\n  branch VARCHAR(255) NOT NULL,\n  test_name VARCHAR(100) NOT NULL,\n  p50 DECIMAL(10, 2),\n  p95 DECIMAL(10, 2),\n  p99 DECIMAL(10, 2),\n  mean DECIMAL(10, 2),\n  min DECIMAL(10, 2),\n  max DECIMAL(10, 2),\n  rps DECIMAL(10, 2),\n  total_requests INT,\n  successful_requests INT,\n  error_rate DECIMAL(5, 2),\n  timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,\n  INDEX idx_commit (commit_sha),\n  INDEX idx_branch (branch),\n  INDEX idx_timestamp (timestamp)\n);\n```\n\n## Workflow Steps\n\n### 1. Code Checkout\n\n```yaml\n- uses: actions/checkout@v4\n  with:\n    fetch-depth: 0  # Full history for comparison\n```\n\nFetches the complete repository history to enable performance comparison with baseline branches.\n\n### 2. Environment Setup\n\n```yaml\n- uses: actions/setup-node@v4\n  with:\n    node-version: '20'\n    cache: 'pnpm'\n```\n\nInstalls Node.js and caches pnpm dependencies for faster builds.\n\n### 3. Database Setup\n\n```yaml\nservices:\n  mysql:\n    image: mysql:8.0\n    env:\n      MYSQL_ROOT_PASSWORD: root\n      MYSQL_DATABASE: test_db\n```\n\nStarts MySQL service for integration tests. Database is automatically initialized.\n\n### 4. Application Build\n\n```yaml\n- run: pnpm build\n```\n\nBuilds the application in production mode to test real-world performance.\n\n### 5. Server Startup\n\n```bash\ntimeout 120 npm run dev > /tmp/server.log 2>&1 &\n# Wait for server to start\nfor i in {1..30}; do\n  if curl -s http://localhost:3000 > /dev/null; then\n    echo \"Server started successfully\"\n    exit 0\n  fi\n  sleep 2\ndone\n```\n\nStarts the development server and waits for it to be ready. Timeout prevents hanging builds.\n\n### 6. Performance Tests\n\n```yaml\n- run: npx artillery run load-tests/baseline.yml -o /tmp/baseline.json\n- run: npx artillery run load-tests/stress.yml -o /tmp/stress.json\n```\n\nRuns baseline and stress tests, saving results as JSON for analysis.\n\n### 7. SLA Validation\n\n```yaml\n- run: node scripts/validate-sla.js /tmp/baseline.json /tmp/stress.json\n```\n\nValidates that all metrics meet SLA thresholds:\n- P50 < 30ms\n- P95 < 50ms\n- P99 < 100ms\n- Error Rate < 0.1%\n\n### 8. Regression Detection\n\n```yaml\n- run: node scripts/compare-performance.js \\\n    --current /tmp/baseline.json \\\n    --baseline main \\\n    --threshold 10\n```\n\nCompares current performance against main branch, detecting >10% regressions.\n\n### 9. Report Generation\n\n```yaml\n- run: npx artillery report /tmp/baseline.json -o /tmp/baseline.html\n- run: npx artillery report /tmp/stress.json -o /tmp/stress.html\n```\n\nGenerates interactive HTML reports with detailed metrics and visualizations.\n\n### 10. Artifact Storage\n\n```yaml\n- uses: actions/upload-artifact@v4\n  with:\n    name: performance-reports\n    path: |\n      /tmp/baseline.json\n      /tmp/stress.json\n      /tmp/baseline.html\n      /tmp/stress.html\n    retention-days: 30\n```\n\nStores test results and reports for 30 days, accessible from workflow run details.\n\n### 11. PR Comment\n\n```yaml\n- uses: actions/github-script@v7\n  with:\n    script: |\n      // Adds performance metrics as PR comment\n```\n\nPosts performance results directly on pull requests for easy review.\n\n### 12. Slack Notification\n\n```yaml\n- uses: slackapi/slack-github-action@v1.24.0\n  with:\n    payload: |\n      {\n        \"text\": \"Performance Test Failed\",\n        ...\n      }\n```\n\nNotifies Slack channel on test failures with links to details.\n\n## Scripts Reference\n\n### validate-sla.js\n\n**Purpose**: Validates performance metrics against SLA thresholds\n\n**Usage**:\n```bash\nnode scripts/validate-sla.js <baseline.json> <stress.json>\n```\n\n**Output**:\n```\n=== Baseline Test (1 RPS) ===\n✓ P50 OK: 15ms < 30ms\n✓ P95 OK: 40ms < 50ms\n✓ P99 OK: 90ms < 100ms\n✓ ERROR RATE OK: 0.05% < 0.1%\n\n=== Stress Test (100 RPS) ===\n✓ P50 OK: 20ms < 30ms\n✓ P95 OK: 48ms < 50ms\n✓ P99 OK: 98ms < 100ms\n✓ ERROR RATE OK: 0.08% < 0.1%\n\n✓ ALL SLA CHECKS PASSED\n```\n\n**Exit Code**:\n- 0: All SLAs passed\n- 1: One or more SLAs violated\n\n### compare-performance.js\n\n**Purpose**: Detects performance regressions vs baseline branch\n\n**Usage**:\n```bash\nnode scripts/compare-performance.js \\\n  --current <file> \\\n  --baseline <branch> \\\n  --threshold <percent>\n```\n\n**Output**:\n```\n=== PERFORMANCE COMPARISON REPORT ===\nThreshold: 10%\n\nCurrent vs Baseline Metrics:\n  P50:  18ms (baseline: 15ms)\n  P95:  45ms (baseline: 40ms)\n  P99:  95ms (baseline: 90ms)\n  Mean: 22ms (baseline: 20ms)\n\n✓ No regressions detected\n\n✓ IMPROVEMENTS DETECTED (1):\n  mean: 20ms (-5.2%)\n```\n\n**Exit Code**:\n- 0: No regressions\n- 1: Regressions detected\n\n### store-metrics.js\n\n**Purpose**: Stores metrics for historical tracking and trend analysis\n\n**Usage**:\n```bash\nnode scripts/store-metrics.js \\\n  --baseline <file> \\\n  --stress <file> \\\n  --commit <sha> \\\n  --branch <name>\n```\n\n**Storage Locations**:\n1. Local: `.performance-metrics/<commit>-<branch>-<timestamp>.json`\n2. Remote: Database (if `METRICS_DB_URL` configured)\n\n## Monitoring and Alerts\n\n### Slack Notifications\n\nAutomatically posts to Slack on:\n- **Test Failures**: SLA violations or errors\n- **Regressions**: Performance degradation > threshold\n- **Success**: Daily scheduled tests pass\n\n**Message Format**:\n```\nPerformance Test Failed 🔴\nRepository: org/repo\nBranch: feature/optimization\nCommit: abc1234\nView Details: [link]\n```\n\n### PR Comments\n\nPerformance metrics posted on every PR:\n\n```markdown\n## Performance Test Results\n\n### Baseline Test (1 RPS)\n- Requests: 30\n- P50: 15ms\n- P95: 40ms\n- P99: 90ms\n- Success Rate: 100.00%\n\n### Stress Test (100 RPS)\n- Requests: 35000\n- P50: 20ms\n- P95: 48ms\n- P99: 98ms\n- Success Rate: 99.95%\n\n[View detailed reports]\n```\n\n## Troubleshooting\n\n### Workflow Fails to Start\n\n**Symptom**: Workflow doesn't appear in Actions tab\n\n**Solution**:\n1. Verify `.github/workflows/performance-tests.yml` is committed\n2. Check branch protection rules don't block workflow\n3. Ensure GitHub Actions is enabled in repository settings\n\n### Server Fails to Start\n\n**Symptom**: \"Server failed to start\" error\n\n**Solution**:\n1. Check `npm run dev` works locally\n2. Verify database is running\n3. Check environment variables are set\n4. Review `/tmp/server.log` for errors\n\n### Artillery Tests Timeout\n\n**Symptom**: Tests hang or timeout\n\n**Solution**:\n1. Increase timeout in workflow (default 30 minutes)\n2. Reduce test duration in `.yml` files\n3. Check server resource limits\n4. Verify network connectivity\n\n### Slack Notifications Not Received\n\n**Symptom**: Workflow succeeds but no Slack message\n\n**Solution**:\n1. Verify `SLACK_WEBHOOK_URL` secret is set\n2. Check webhook URL is valid\n3. Ensure Slack app has permission to post\n4. Review workflow logs for webhook errors\n\n## Best Practices\n\n1. **Run tests regularly**: Daily schedule catches regressions early\n2. **Set appropriate thresholds**: 10% regression threshold balances sensitivity\n3. **Monitor trends**: Track metrics over time to identify patterns\n4. **Review failures**: Investigate every SLA violation\n5. **Optimize before release**: Fix regressions before merging to main\n6. **Document baselines**: Keep records of expected performance\n7. **Test realistically**: Use production-like data and scenarios\n8. **Automate alerts**: Notify team immediately on violations\n\n## Advanced Configuration\n\n### Custom Thresholds\n\nModify SLA thresholds in `scripts/validate-sla.js`:\n\n```javascript\nconst SLA = {\n  p50: 30,      // 30ms\n  p95: 50,      // 50ms\n  p99: 100,     // 100ms\n  errorRate: 0.1, // 0.1%\n};\n```\n\n### Custom Notifications\n\nAdd custom notification logic in workflow:\n\n```yaml\n- name: Notify Teams\n  if: failure()\n  run: |\n    curl -X POST ${{ secrets.TEAMS_WEBHOOK_URL }} \\\n      -H 'Content-Type: application/json' \\\n      -d '{\"text\": \"Performance test failed\"}'\n```\n\n### Performance Budgets\n\nImplement performance budgets by comparing against targets:\n\n```bash\nnode scripts/validate-budget.js \\\n  --current /tmp/baseline.json \\\n  --budget 50ms  # P95 budget\n```\n\n## References\n\n- [GitHub Actions Documentation](https://docs.github.com/en/actions)\n- [Artillery Documentation](https://artillery.io/docs)\n- [Performance Monitoring Best Practices](https://web.dev/performance/)\n- [SLA Definition Guide](https://en.wikipedia.org/wiki/Service-level_agreement)\n
