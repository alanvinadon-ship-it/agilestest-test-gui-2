# Performance Testing Guide — Multi-Engine AI Routing\n\nComprehensive guide for load testing and performance validation of the multi-engine AI routing system using Artillery.\n\n## Overview\n\nThis guide covers automated performance testing to validate engine resolution latency under various load conditions. The test suite ensures the system meets SLA targets:\n\n- **P50 (Median)**: < 30ms\n- **P95 (95th percentile)**: < 50ms\n- **P99 (99th percentile)**: < 100ms\n- **Error Rate**: < 0.1%\n\n## Test Scenarios\n\n### 1. Baseline Test (`baseline.yml`)\n\n**Purpose**: Establish baseline performance with minimal load\n\n**Configuration**:\n- Duration: 30 seconds\n- Arrival Rate: 1 request per second\n- Total Requests: ~30\n- Concurrent Users: 1\n\n**Scenarios**:\n1. **Engine Resolution**: List all engines for organization\n2. **Routing Rule Resolution**: List routing rules for use case\n3. **Primary Engine Lookup**: Fetch primary engine only\n\n**Expected Results**:\n- P50: 10-15ms\n- P95: 20-25ms\n- P99: 30-40ms\n- Success Rate: 100%\n\n**Run Command**:\n```bash\nartillery run load-tests/baseline.yml\n```\n\n### 2. Stress Test (`stress.yml`)\n\n**Purpose**: Validate performance under sustained load\n\n**Configuration**:\n- Phase 1 (Ramp-up): 10 → 100 RPS over 60 seconds\n- Phase 2 (Sustained): 100 RPS for 5 minutes (300 seconds)\n- Phase 3 (Ramp-down): 100 → 0 RPS over 60 seconds\n- Total Duration: ~7 minutes\n- Total Requests: ~35,000\n- Peak Concurrent Users: ~100\n\n**Scenarios** (weighted distribution):\n1. **Engine Resolution** (40%): Primary operation\n2. **Routing Rule Resolution** (35%): Secondary operation\n3. **Primary Engine Lookup** (25%): Fallback operation\n\n**Expected Results**:\n- P50: 15-20ms\n- P95: 40-50ms\n- P99: 80-100ms\n- Success Rate: > 99.9%\n- Error Rate: < 0.1%\n\n**Run Command**:\n```bash\nartillery run load-tests/stress.yml\n```\n\n### 3. Spike Test (`spike.yml`)\n\n**Purpose**: Test system resilience to sudden traffic spikes\n\n**Configuration**:\n- Phase 1 (Baseline): 10 RPS for 60 seconds\n- Phase 2 (Spike): 10 → 150 RPS in 10 seconds\n- Phase 3 (Sustained): 150 RPS for 60 seconds\n- Phase 4 (Recovery): 150 → 10 RPS in 30 seconds\n- Phase 5 (Baseline): 10 RPS for 60 seconds\n- Total Duration: ~4 minutes\n- Total Requests: ~10,000\n- Peak Concurrent Users: ~150\n\n**Scenarios** (equal distribution):\n1. **Engine Resolution** (50%)\n2. **Routing Resolution** (50%)\n\n**Expected Results**:\n- P50: 20-30ms (during spike)\n- P95: 50-80ms (during spike)\n- P99: 100-150ms (during spike)\n- Success Rate: > 99%\n- Recovery Time: < 30 seconds after spike\n\n**Run Command**:\n```bash\nartillery run load-tests/spike.yml\n```\n\n## Running Tests\n\n### Prerequisites\n\n1. **Install Artillery**:\n   ```bash\n   npm install -D artillery\n   # or\n   pnpm add -D artillery\n   ```\n\n2. **Start the server**:\n   ```bash\n   npm run dev\n   # Server should be running on http://localhost:3000\n   ```\n\n3. **Verify server is accessible**:\n   ```bash\n   curl http://localhost:3000\n   ```\n\n### Run All Tests\n\n```bash\ncd load-tests\n./run-tests.sh\n```\n\nThis script:\n1. Verifies server is running\n2. Runs baseline test\n3. Runs stress test\n4. Runs spike test\n5. Saves results to `results/` directory\n6. Prints summary statistics\n\n### Run Individual Tests\n\n```bash\n# Baseline\nartillery run load-tests/baseline.yml\n\n# Stress\nartillery run load-tests/stress.yml\n\n# Spike\nartillery run load-tests/spike.yml\n```\n\n### Run with Output File\n\n```bash\nartillery run load-tests/baseline.yml -o results/baseline.json\nartillery run load-tests/stress.yml -o results/stress.json\nartillery run load-tests/spike.yml -o results/spike.json\n```\n\n### Generate HTML Reports\n\n```bash\nartillery report results/baseline.json\nartillery report results/stress.json\nartillery report results/spike.json\n```\n\nThis generates `results/baseline.html`, `results/stress.html`, and `results/spike.html` with detailed visualizations.\n\n## Understanding Results\n\n### Metrics Explained\n\n**Latency Percentiles**:\n- **Min**: Fastest request\n- **P50 (Median)**: 50% of requests are faster than this\n- **P95**: 95% of requests are faster than this (SLA target: < 50ms)\n- **P99**: 99% of requests are faster than this (SLA target: < 100ms)\n- **Max**: Slowest request\n- **Avg**: Average latency\n\n**Throughput**:\n- **RPS (Requests Per Second)**: Number of requests completed per second\n- **Concurrency**: Number of simultaneous connections\n\n**Success Metrics**:\n- **Success Rate**: Percentage of requests that returned HTTP 2xx\n- **Error Rate**: Percentage of failed requests (HTTP 4xx, 5xx, timeouts)\n\n### SLA Validation\n\nThe processor validates SLAs during test execution:\n\n```javascript\nconst p50Threshold = 30;   // 30ms\nconst p95Threshold = 50;   // 50ms\nconst p99Threshold = 100;  // 100ms\n```\n\nViolations are logged with:\n- Request ID\n- Actual latency\n- Threshold exceeded\n- Timestamp\n\n## Performance Optimization\n\nIf tests show SLA violations:\n\n### 1. Database Query Optimization\n\n```typescript\n// Check indexes on aiEngines and aiRoutingRules\nSELECT * FROM aiEngines WHERE orgId = ? AND enabled = true;\nSELECT * FROM aiRoutingRules WHERE orgId = ? AND useCase = ? ORDER BY priority;\n```\n\n**Ensure indexes exist**:\n```sql\nCREATE INDEX idx_ai_engines_org_enabled ON aiEngines(orgId, enabled);\nCREATE INDEX idx_ai_routing_org_usecase ON aiRoutingRules(orgId, useCase, priority);\n```\n\n### 2. Caching Strategy\n\nImplement Redis caching for frequently accessed data:\n\n```typescript\n// Cache engine list for 60 seconds\nconst cacheKey = `engines:${orgId}`;\nlet engines = await redis.get(cacheKey);\nif (!engines) {\n  engines = await db.select().from(aiEngines).where(eq(aiEngines.orgId, orgId));\n  await redis.set(cacheKey, JSON.stringify(engines), 'EX', 60);\n}\n```\n\n### 3. Connection Pooling\n\nEnsure database connection pool is properly configured:\n\n```typescript\nconst db = drizzle(process.env.DATABASE_URL, {\n  casing: 'snake_case',\n  pool: {\n    min: 5,\n    max: 20,  // Increase for high concurrency\n  },\n});\n```\n\n### 4. API Response Optimization\n\nMinimize response payload:\n\n```typescript\n// Only return necessary fields\nawait db.select({\n  uid: aiEngines.uid,\n  name: aiEngines.name,\n  provider: aiEngines.provider,\n  enabled: aiEngines.enabled,\n}).from(aiEngines).where(eq(aiEngines.orgId, orgId));\n```\n\n## Monitoring in Production\n\n### Key Metrics to Track\n\n1. **Engine Resolution Latency**\n   - P50, P95, P99 percentiles\n   - Alert if P95 > 50ms\n   - Alert if P99 > 100ms\n\n2. **Error Rate**\n   - Alert if > 0.1%\n   - Track by error type (timeout, 5xx, etc.)\n\n3. **Throughput**\n   - RPS by scenario\n   - RPS by organization\n\n4. **Resource Utilization**\n   - CPU usage\n   - Memory usage\n   - Database connection pool utilization\n   - Network I/O\n\n### Prometheus Metrics\n\nExpose metrics for Prometheus scraping:\n\n```typescript\nimport { register, Counter, Histogram } from 'prom-client';\n\nconst engineResolutionLatency = new Histogram({\n  name: 'engine_resolution_latency_ms',\n  help: 'Engine resolution latency in milliseconds',\n  buckets: [10, 20, 30, 50, 100, 200, 500],\n  labelNames: ['orgId', 'useCase'],\n});\n\nconst engineResolutionErrors = new Counter({\n  name: 'engine_resolution_errors_total',\n  help: 'Total engine resolution errors',\n  labelNames: ['orgId', 'errorType'],\n});\n```\n\n### Grafana Dashboards\n\nCreate dashboards to visualize:\n- Engine resolution latency (P50, P95, P99)\n- Error rate by organization\n- Throughput (RPS) over time\n- Resource utilization\n- Cache hit rate\n\n## Troubleshooting\n\n### High Latency (P95 > 50ms)\n\n**Symptoms**: Tests show P95 latency exceeding 50ms\n\n**Investigation**:\n1. Check database query performance: `EXPLAIN ANALYZE SELECT ...`\n2. Verify indexes exist and are being used\n3. Check database connection pool utilization\n4. Monitor CPU and memory usage\n5. Check network latency to database\n\n**Solutions**:\n1. Add missing indexes\n2. Optimize queries (select only needed columns)\n3. Implement caching\n4. Increase connection pool size\n5. Scale database vertically or horizontally\n\n### High Error Rate (> 0.1%)\n\n**Symptoms**: Tests show error rate exceeding 0.1%\n\n**Investigation**:\n1. Check server logs for errors\n2. Verify database connectivity\n3. Check for timeout errors\n4. Monitor resource exhaustion (connections, memory)\n\n**Solutions**:\n1. Fix application errors\n2. Increase database connection pool\n3. Increase request timeout\n4. Scale horizontally\n\n### Slow Recovery from Spike\n\n**Symptoms**: After spike, latency remains high for extended period\n\n**Investigation**:\n1. Check for connection pool exhaustion\n2. Monitor garbage collection pauses\n3. Check for database lock contention\n\n**Solutions**:\n1. Increase connection pool size\n2. Tune garbage collection\n3. Optimize long-running queries\n4. Implement circuit breaker pattern\n\n## CI/CD Integration\n\n### GitHub Actions Workflow\n\n```yaml\nname: Performance Tests\n\non:\n  push:\n    branches: [main, develop]\n  pull_request:\n    branches: [main]\n\njobs:\n  performance:\n    runs-on: ubuntu-latest\n    services:\n      mysql:\n        image: mysql:8.0\n        env:\n          MYSQL_ROOT_PASSWORD: root\n    steps:\n      - uses: actions/checkout@v3\n      - uses: actions/setup-node@v3\n        with:\n          node-version: '20'\n      - run: npm install\n      - run: npm run build\n      - run: npm run dev &\n      - run: sleep 5  # Wait for server to start\n      - run: npx artillery run load-tests/baseline.yml -o results/baseline.json\n      - run: npx artillery report results/baseline.json\n      - uses: actions/upload-artifact@v3\n        with:\n          name: performance-reports\n          path: results/\n```\n\n## Best Practices\n\n1. **Run tests regularly**: Daily or on every commit to catch regressions\n2. **Establish baselines**: Know your system's normal performance\n3. **Test realistically**: Use realistic data volumes and query patterns\n4. **Monitor trends**: Track performance over time\n5. **Automate alerts**: Alert on SLA violations\n6. **Document results**: Keep historical records for analysis\n7. **Test before deployment**: Always validate performance before production release\n8. **Load test new features**: Test new features under load before release\n\n## References\n\n- [Artillery Documentation](https://artillery.io/docs)\n- [Load Testing Best Practices](https://artillery.io/blog/load-testing-best-practices)\n- [Performance Optimization Guide](https://www.postgresql.org/docs/current/performance.html)\n- [Database Indexing](https://use-the-index-luke.com/)\n
